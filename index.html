Challenges

There are a lot of problems to solve and aspects to consider when designing plugin systems and more generally, component architectures. Application Domain Conformance, Encapsulation, Modularity, Communication, Security, Loose Coupling, Deployment and Testability to name a few. Today we will focus on a subset that troubled us the most:

Engage everyone

Asking questions to make a lecture interactive can be done also without technology. However, with technology everyone can answer at the same time. So as a teacher, how do you deal with hundreds of students sending you their programming code solutions? ASQ can help to aggregate and summarize the answers so they can be discussed with the audience.

Empower teachers

We want to enable these things while keeping the cognitive overhead as low as possible for the presenter. This is where clustering submissions and visualizations come into play.

Real classroom

  • participation was optional
  • students’ identity concealed
  • learning set-up equal for all participants.
Previous works Methodological limitations:
  • experiments took place in a controlled setting
  • where the students’ identities were known
  • explicitly assigned to experimental conditions that could have harmed their learning experience

Scalability & availability

Privacy & fairness

These issues of privacy and fairness are not compatible with the requirements of a real-world course, where students must be guaranteed equal treatment, and privacy must be preserved.

So how to we engage students to educational interactions? We have slides that can have questions for which the users give answers. In more detail, ASQ is a Web platform for interactive HTML slides, that allows you to interleave questions in them and receive answers from your audience in real time. Admittedly, the main job of a presenter is to engage their audience. And we use a lot of ways to do that. Assume for example that I wanted to find out our favourite programming language. I could do a show of hands or a voice vote. Who likes Python?... Who likes Javascript... But this is hard to scale or accurately measure. Or I could just do this..

No Context Switch

with ASQ we make it easy because there is no switch between slides, questions and answers (repeat the logo name definition) everything is integrated. This reduces opportunities for off-task behavior and removes the cognitive load associated with switching between tools

Recognition Cued/free Recall Live-Programming
We support .different question types ranging from simple multi-choice and text input questions; to labeling, ordering and highlight text to more open ended question types like Javascript programming or SQL queries that run in a database inside the browser.

The ASQ cycle is the teaching model we like to follow in class and ASQ is optimized around that. It's based on the premises that short teaching segments followed by recall segements allow for better learining. The way it works is as follows. We present a short segment and then pose associated questions to the classroom. Students answer and we aggregated and assess/cluster the answers. Then we discuss results with the students to clear misconceptions, fill in knowledge gaps and comment on the merits of different types of answers. And then we start all over again.
Here's this cycle at play at TU Delft. Students are answers a SQLite questions
The presenter tracks the progress and can peek into the answers
Then he moves on to discuss particular solutions. Notice that he can execute each student submission in his own question instance and discuss the result of the query

Architecture

[Triglianos2014ICWE]
ASQ is a liquid application. Web application with distributed UI running across multiple devices. We have different roles like presenters and viewers and different views like beamer and cockpit.
[Triglianos2015ICWE]
Each ASQ plugin can have both serverside and clientside logic. The plugin systems of ASQ allows to use a single code base for both and offers an API for communication and interaction with the core.

Requirements

  • Validate presentations, remove solutions, testing and assessing answers, giving feedback to students
  • Encapsulation
    • CSS: Base rule changes, Naming collisions, subtree matches
    • JavaScript: global namespace pollution, memory leaks
  • Communication
    • Serverside
    • Clientside
    • inter-plugin (in-between client-side or server-side components of different plugins)
    • intra-plugin (between client-side and server-side components)
  • Deployment and Publication
    • Single code repository
    • Easily isolate and deploy frontend and backend code

Serverside architecture

One interesting implementation detail that decouples the core from plugins and can isolate runnning plugins in case of failure is the proxy object. A proxy object is a single façade interface between plugin modules and the host system. It exposes the core API to plugins. Each plugin gets instantiated with one from the core.
